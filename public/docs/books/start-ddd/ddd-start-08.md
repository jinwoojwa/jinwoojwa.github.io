## 애그리거트와 트랜잭션

동시에 여러 사용자가 하나의 애그리거트를 수정하면 일관성 문제가 발생할 수 있음.

```text
사용자 A ──┐
         ├── Order 수정
사용자 B ──┘
```

### 해결 방법

- 비관적 락(Pessimistic Lock)
- 낙관적 락(Optimistic Lock)

---

## 비관적 락

수정하는 동안 다른 트랜잭션의 접근 차단

```text
User A
   │
   ▼
Lock 획득
   │
   ▼
Order 수정
   │
   ▼
Commit
   │
   ▼
Lock 해제

──────────────

User B
   │
   ▼
대기(Blocking)
```

### 특징

- DB Lock 사용 (`SELECT ... FOR UPDATE`)
- 데이터 충돌 방지
- 다른 트랜잭션은 대기

### 단점

- 성능 저하
- 데드락 발생 가능

> **NOTE**
>
> 충돌이 자주 발생하는 경우 비관적 락이 적합.

---

## 낙관적 락

수정을 막지 않고 저장 시점에 충돌 검사

```text
Order
 ├── id
 ├── ...
 └── version
```

```text
조회(version=3)
       │
       ▼
      수정
       │
       ▼
UPDATE ... WHERE version = 3
       │
       ▼
성공 → version = 4

또는

실패 → OptimisticLockException
```

### 특징

- `@Version` 사용
- 충돌 시 예외 발생
- 재시도 로직 필요

> **NOTE**
>
> 충돌이 자주 발생하지 않는 경우
> 낙관적 락이 성능상 유리.

---

## 애그리거트와 버전

애그리거트 내부 객체가 변경되어도 애그리거트 전체가 변경된 것으로 간주

필요한 경우 **강제 버전 증가** 사용

```text
Order (version = 3)
│
├── OrderItem 수정
└── Delivery 수정

↓

Order version = 4
```

예)

```java
LockModeType.OPTIMISTIC_FORCE_INCREMENT
```

---

## 비관적 오프라인 락

여러 트랜잭션에 걸쳐 잠금 유지

### 특징

- 일반적인 DB Lock과 다름
- LockManager 등을 이용하여 구현

> **NOTE**
>
> 실무에서는 비관적 락과 낙관적 락을 주로 사용하며,
> 오프라인 락은 특수한 경우에만 사용.

## 비관적 락 vs 낙관적 락

| 항목        | 비관적 락 (Pessimistic Lock)   | 낙관적 락 (Optimistic Lock) |
| ----------- | ------------------------------ | --------------------------- |
| 동작 방식   | 수정 전 미리 잠금              | 저장 시 충돌 여부 확인      |
| DB Lock     | 사용 (`SELECT ... FOR UPDATE`) | 사용하지 않음               |
| 충돌 처리   | 충돌 자체를 방지               | 충돌 발생 후 예외 처리      |
| 대기 여부   | 다른 트랜잭션 대기             | 대기 없음                   |
| 성능        | 상대적으로 낮음                | 상대적으로 높음             |
| 장점        | 데이터 정합성 보장             | 동시성 처리에 유리          |
| 단점        | 데드락, 성능 저하 가능         | 재시도 로직 필요            |
| 적합한 경우 | 충돌이 자주 발생               | 충돌이 드물게 발생          |

> **NOTE**
>
> 대부분의 일반적인 서비스에서는 **낙관적 락을 우선 고려**하고,
> 재고 차감처럼 **충돌 가능성이 높은 경우** 비관적 락을 사용하는 경우가 많음.
