## 도메인이란

> **도메인**: 소프트웨어로 해결하고자 하는 문제 영역

## 도메인 모델

**도메인 모델**은 특정 도메인을 개념적으로 표현한 것이다.

- 표현 방법은 다양하다 → 객체 이용, 상태 다이어그램 이용, 그래프 등
- 도메인 자체를 이해하기 위한 **개념 모델**로 시작한다
  - 개념 모델: 문제를 분석한 결과물
  - 이를 구현 가능한 형태로 전환한 것이 **구현 모델**
- 처음부터 완벽한 개념 모델을 설계하는 것은 불가능하다
  → 구현 과정에서 점진적으로 발전시켜야 한다 (구현하면서 도메인에 대한 이해도가 높아짐)

## 아키텍처 구성

도메인 모델 패턴을 적용한 전형적인 계층 구조는 다음과 같다.

```
표현 ---> 응용 ---> 도메인 ---> 인프라
```

| 계층   | 역할                                                             |
| ------ | ---------------------------------------------------------------- |
| 표현   | 사용자 요청 처리, 정보를 보여줌                                  |
| 응용   | 요청한 기능 수행, 업무 로직은 직접 구현하지 않음                 |
| 도메인 | 도메인 계층을 조합해 기능 수행, 시스템이 제공할 도메인 규칙 구현 |
| 인프라 | DB, 메시징 시스템 등 외부 시스템과의 연동 처리                   |

## 도메인 모델 도출

도메인 모델링을 통해 핵심 구성요소, 규칙, 기능을 도출한다.

**예시: 주문 도메인 요구사항**

- 출고를 하면 배송지 변경이 불가능하다
- 출고 전에만 주문 취소가 가능하다

위 요구사항으로부터 다음 기능을 도출할 수 있다.

- 출고 상태 변경
- 배송지 정보 변경
- 주문 취소

> 요구사항으로부터 도메인 모델을 이끌어내며, 코드 자체도 하나의 문서다. 도메인 지식이 잘 묻어나도록 코드를 작성해야 한다.

## 엔티티와 밸류

### 엔티티

**엔티티**는 고유한 식별자를 가진다. 식별자 생성 방법은 다음과 같다.

- 특정 규칙 사용: UUID, ULID, NanoID 등 고유 식별자 생성기
- 직접 입력: 아이디, 이메일 등
- 일련번호 사용: 시퀀스 또는 DB 자동 증가 컬럼

### 밸류 타입

**밸류 타입**은 개념적으로 완전한 하나를 표현한다.

```java
// Before
String address;
String zipcode;

// After: 밸류 타입으로 묶기
class Address {
    String address;
    String zipcode;
}

Address address;
```

```java
// Before
int price;

// After: 밸류 타입으로 감싸기
class Money {
    private int value;
}

Money price;
```

- 밸류 타입에 관련 기능을 추가할 수 있다
- 코드의 의미를 더 잘 이해할 수 있도록 돕는다
- **불변 객체**로 구현하는 것이 좋다 → 참조 투명성 확보, 스레드에 안전

## Setter 사용 지양하기

Setter를 남용하면 다음과 같은 문제가 생긴다.

- 도메인의 핵심 개념과 의도가 사라진다
- 도메인 객체가 생성 시점에 완전한 상태를 갖추지 못하게 된다

> 대안: 생성 시점에 필요한 정보를 모두 전달한다 (생성자 활용)

## 도메인 용어와 유비쿼터스 언어

- 도메인에서 사용하는 용어를 코드에 그대로 반영해야 한다
- **유비쿼터스 언어**: 전문가, 관계자, 개발자가 도메인과 관련된 공통의 언어를 만들고, 모든 곳에서 동일하게 사용하는 것
- 시간이 지날수록 도메인에 대한 이해도가 높아진다 → 용어를 갱신하고 문서화해야 한다
