## 리포지터리

- 리포지터리는 애그리거트 단위로 조회 및 저장 담당

### 위치

- 리포지터리 인터페이스 → 도메인 영역
- 리포지터리 구현체 → 인프라 영역

![패키지 구성](/assets/images/repository_domain.png)

> **NOTE**
>
> **도메인은 인터페이스만 알고, 구현체는 인프라에서 제공함.**
>
> 이를 통해 구현 기술과 도메인을 분리함.

---

## 애그리거트 로딩 전략

- 애그리거트는 **사용하는 시점에는 완전한 상태**여야 함.

```java
// 애그리거트 루트인 Product에 속한 모든 객체가 완전한 상태여야 함
Product product = productRepository.findById(productId);
```

### 즉시 로딩(EAGER)

- 연관 객체를 함께 조회
- 애그리거트를 한 번에 구성

### 문제점

- 조인 증가
- 카테시안 곱(Cartesian Product) 발생 가능
- 조회 성능 저하

> **NOTE**
>
> **애그리거트는 개념적으로 하나이지만, 항상 모든 객체를 한 번에 로딩할 필요는 없음.**

### 언제 완전한 상태가 필요한가?

- 상태 변경
- 화면 표시

조회만 필요한 경우에는 조회 전용 모델(CQRS)을 사용하는 것이 더 적합할 수 있음.

## 애그리거트의 영속성

- 애그리거트는 하나의 단위로 저장 및 삭제

```text
Order (Root)
 ├── OrderItem
 ├── Delivery
 └── Payment
```

- 관련 객체도 함께 영속화
- `Cascade` 사용
- 하나의 트랜잭션으로 처리

## 식별자 생성

### 생성 방식

- 사용자 직접 생성
- 도메인 로직으로 생성
- DB 자동 생성(AUTO_INCREMENT, SEQUENCE 등)

### DB 자동 생성

```text
객체 생성
    │
    ▼
id 없음
    │
    ▼
INSERT
    │
    ▼
id 생성
```

- DB에서 식별자를 생성하는 경우 **저장 이후에 ID를 획득**할 수 있음.

## JPA와 DIP

원칙적으로는 DIP를 완전히 지키지 못함.

### 이유

- 엔티티가 JPA 애너테이션 사용

```java
@Entity
@Table
@Id
@Column
```

- Repository가 JPA Repository 상속

↓

- 도메인이 JPA에 의존

> **NOTE**
>
> **DDD에서는 DIP를 지향하지만, 실무에서는 생산성과 유지보수를 위해 JPA에 일부 의존하는 타협을 많이 선택함.**
