## 1. Vue Router: 페이지 이동을 위한 내비게이션

Vue Router는 사용자가 URL을 입력하거나 클릭할 때, 페이지를 새로고침하지 않고 컴포넌트를 교체하여 화면을 전환하는 라이브러리이다.

- **기본 구성:** URL 경로와 컴포넌트를 매핑하는 라우트 객체를 생성한다.
- **핵심 컴포넌트:**
  - `<router-link>`: 페이지 이동을 위한 앵커 태그 역할.
  - `<router-view>`: 현재 경로에 해당하는 컴포넌트가 렌더링되는 영역.

<br>

**1️⃣ 라우트 설정 (index.js)**

가장 먼저 어떤 URL 경로에 어떤 컴포넌트를 보여줄지를 설정해야 한다.

이를 위해 routes 배열에 경로와 컴포넌트를 매핑한다.

```javascript
// router/index.js
import { createRouter, createWebHistory } from 'vue-router';
import HomeView from '../views/HomeView.vue';
import AboutView from '../views/AboutView.vue';

const routes = [
  { path: '/', component: HomeView },
  { path: '/about', component: AboutView },
];

const router = createRouter({
  history: createWebHistory(),
  routes,
});
export default router;
```

<br>

**2️⃣ Vue 인스턴스에 라우터 연결 (main.js)**

설정한 라우터 객체를 Vue 애플리케이션(`app`)에 주입해야 한다.

그래야 앱 전체에서 라우터 기능을 사용할 수 있다.

```javascript
// main.js
import { createApp } from 'vue';
import App from './App.vue';
import router from './router'; // 위에서 만든 설정 파일을 가져옴

const app = createApp(App);

app.use(router); // 라우터를 앱에 등록
app.mount('#app');
```

<br>

**3️⃣ 템플릿에 렌더링하기 (App.vue)**

이제 `<router-link>`로 이동 버튼을 만들고, `<router-view>`를 통해 해당 경로의 컴포넌트가 교체되어 나타나게 한다.

```html
<template>
  <nav>
    <router-link to="/">홈으로</router-link>
    <router-link to="/about">소개 페이지</router-link>
  </nav>

  <router-view />
</template>
```

<br>

### 💡 Vue 라우터가 화면을 바꾸는 과정

Vue Router를 통해 구현된 화면 전환은 다음과 같은 단계로 이루어진다.

1. **경로 매핑 (Route Mapping):** `router/index.js`에서 URL 경로(`path`)와 해당 경로에 대응하는 컴포넌트(`component`)를 정의한다. 이 설정은 라우터 객체가 어떤 URL에서 어떤 UI를 렌더링해야 할지 판단하는 기준이 된다.

2. **라우터 주입 (Plugin Registration):** `main.js`에서 `app.use(router)`를 실행함으로써, Vue 애플리케이션 전역에서 `useRoute()`, `useRouter()`를 통해 접근할 수 있게 된다. 이를 통해 애플리케이션 어디서든 현재 경로를 조회하거나 경로를 변경하는 제어가 가능해진다.

3. **동적 컴포넌트 교체 (Dynamic Component Injection):** 사용자가 `<router-link>`를 통해 경로를 변경하면, 라우터는 URL 변경(History API)을 트리거로 라우트 매칭을 수행하고, 해당 경로에 맞는 컴포넌트를 결정한다. 이후 `<router-view>` 영역에 해당 컴포넌트를 동적으로 마운트하여 화면을 갱신한다.

이처럼 Vue Router는 전체 페이지를 새로고침하는 전통적인 방식과 달리, **브라우저의 히스토리 API와 Vue의 반응형 시스템을 결합** 하여 필요한 컴포넌트만 효율적으로 교체하는 SPA(Single Page Application) 환경을 제공한다.

## 2. Pinia: 전역 상태 관리

애플리케이션이 커지면 `props`와 `emit`만으로는 데이터 전달이 복잡해진다. `Pinia`는 앱 어디서든 접근할 수 있는 '공용 저장소' 를 제공한다.

**핵심 구성:**

- `state`: 저장소의 데이터(반응형 변수).

- `getters`: 상태를 계산된 값으로 반환(computed와 동일).

- `actions`: 상태 변경과 비즈니스 로직을 담당하는 함수. 비동기 처리가 가능하며, 컴포넌트의 methods와 역할이 유사

```javascript
// useCounterStore.js
import { ref, computed } from 'vue';
import { defineStore } from 'pinia';

export const useCounterStore = defineStore('counter', () => {
  const count = ref(0);
  const doubleCount = computed(() => count.value * 2);
  function increment() {
    count.value++;
  }

  return { count, doubleCount, increment };
});
```

<br>

### 💡 TIP: 왜 state를 직접 수정하지 않을까?

Pinia에서 `state`를 컴포넌트에서 직접 변경할 수도 있지만, 가급적 **actions**를 거치는 것을 권장한다.

- **데이터 무결성:** 여러 컴포넌트가 제각각 상태를 수정하면 버그를 찾기 힘들다. `actions`에 수정 로직을 하나로 모아두면, 데이터 변경의 경로가 단일화되어 유지보수가 훨씬 수월하다.
- **예측 가능성:** "데이터가 변할 때 반드시 특정 로직(예: 서버 저장)이 수행되어야 한다"는 규칙을 강제할 수 있다.

<br>

**1️⃣ 좋지 않은 예: 컴포넌트에서 상태를 직접 수정**

데이터 변경의 원인을 찾기 어렵고, 변경 시 필요한 부가 작업(서버 저장 등)을 빼먹을 위험이 있다.

```javascript
// 컴포넌트 코드
const store = useCounterStore();

// 아무 곳에서나 직접 수정 가능 (버그 추적 힘듦)
store.count++;
```

<br>

**2️⃣ 권장하는 예: actions를 통한 상태 변경**

데이터 변경의 경로가 단일화되어, 나중에 '로그 기록' 이나 '서버 연동' 같은 로직을 추가하기 매우 유리하다.

```javascript
// store/counter.js (Pinia 정의)
export const useCounterStore = defineStore('counter', () => {
  const count = ref(0);

  // 상태를 수정하는 로직을 actions에 집중
  function increment() {
    // 1. 상태 변경 전 부가 작업 가능 (예: 서버 API 호출, 로깅)
    console.log('카운트 증가 시도: ', count.value);

    // 2. 실제 상태 변경
    count.value++;
  }

  return { count, increment };
});

// 컴포넌트 코드
const store = useCounterStore();

// 반드시 정의된 메서드를 호출해야 함
store.increment();
```

<br>

### 💡 데이터 관리의 위계 (선택 가이드)

애플리케이션이 성장할 때, 모든 데이터를 Pinia에 넣으면 오히려 관리 포인트가 늘어난다. 다음 기준을 따라 위계를 세우는 것이 좋다.

1. **로컬 데이터 (Props/Emit):** 현재 컴포넌트 안에서만 의미가 있는 값인가? → **Props**
2. **범위 데이터 (Provide/Inject):** 현재 컴포넌트와 그 자식들만 공유하면 되는가? → **Provide/Inject**
3. **핵심 데이터 (Pinia):** 여러 페이지에서 공통으로 사용하며, 서버와 동기화가 필요한가? → **Pinia**

<br><br>

## 3. 데이터 관리 도구 선택 방법

| 도구               | 추천 상황                                                   |
| :----------------- | :---------------------------------------------------------- |
| **Props/Emit**     | 부모-자식 간의 1:1 관계 데이터 이동                         |
| **Provide/Inject** | 깊게 중첩된 컴포넌트 간의 고정된 전역 데이터 (테마 등)      |
| **Pinia**          | 앱 전체에서 빈번하게 수정되고 공유되는 핵심 비즈니스 데이터 |

<br><br>

## 📚 정리

- `Vue Router`는 `SPA`의 핵심으로, 페이지 새로고침 없이 사용자 경험을 매끄럽게 전환한다.

- `Pinia`는 복잡해지는 상태 관리를 한곳으로 모아 효율적이고 예측 가능한 데이터 흐름을 제공한다.

- **핵심:** 무조건 `Pinia`를 쓰기보다, 컴포넌트 내부에서 해결 가능한 것은 `Props`로, 전역적으로 쓰이는 데이터는 `Pinia`로 분리하는 **'데이터의 위계'** 를 세우는 것이 중요하다.
