## 1. Provide/Inject: 프롭 드릴링의 해결

컴포넌트 트리가 깊어지면, 부모에서 아주 깊은 자식 컴포넌트로 데이터를 전달하기 위해 중간 단계의 컴포넌트들이 불필요하게 Props를 전달해야 한다.

이를 **'프롭 드릴링(Prop Drilling)'** 이라 한다.

하지만, `provide`와 `inject`를 사용하면 중간 단계를 건너뛰고 데이터를 즉시 주입할 수 있다.

```javascript
// 부모 컴포넌트 (데이터 제공)
<script setup>
import { provide } from 'vue'

provide('theme', 'dark')
</script>

// 깊은 자식 컴포넌트 (데이터 수신)
<script setup>
import { inject } from 'vue'

const theme = inject('theme')
</script>
```

<br>

### 💡 TIP: 안전한 데이터 수정 (Injection Key)

단순한 값 전달을 넘어, 상태를 수정하는 함수까지 제공하면 부모의 데이터 주도권을 유지하며 자식과 안전하게 소통할 수 있다.

다시 말하면, 자식은 부모의 상태를 부모가 정한 방식대로만 안전하게 수정할 수 있게 된다.

```javascript
// 부모: 데이터와 수정 기능을 묶어서 제공
provide('userState', {
  user,
  updateName: (newName) => {
    user.name = newName;
  },
});

// 자식: 제공받은 수정 기능만 사용
const { user, updateName } = inject('userState');
```

<br>

### 💡 Props 대신 Provide/Inject를 쓰는 게 더 좋은가?

결론부터 말하자면 **아니다.**

대부분의 경우 **Props를 사용하는 것이 정석이다.** 그 이유는 다음과 같다.

- **데이터의 흐름 추적 (Props의 장점):** `Props`는 코드를 보면 어느 부모 컴포넌트가 이 데이터를 자식에게 전달했는지 한눈에 알 수 있다. 반면, `provide/inject`는 데이터가 어디서 제공되었는지 직관적으로 찾기 어려워, 규모가 큰 애플리케이션에서는 데이터의 흐름을 쫓기가 힘들 수 있다.

- **재사용성 (Props의 장점):** `Props`를 받는 컴포넌트는 "데이터만 넣어주면 어디서든 작동하는" 독립적인 부품이 된다. 하지만 `inject`를 사용하는 컴포넌트는 특정 부모가 데이터를 제공해주지 않으면 작동하지 않는 '의존적인 부품'이 된다.

<br>

**❓ 언제 무엇을 써야 하는가?**

- `Props`: 대부분의 일반적인 부모-자식 관계 (90% 이상)

- `Provide/Inject`: 아주 깊게 중첩된 컴포넌트 구조에서 테마(색상), 사용자 정보, 다국어 설정처럼 **앱 전체에서 공통으로 공유해야 하는 '전역적인 데이터'** 를 전달할 때

요약하자면, 컴포넌트끼리 직접적인 부모-자식 관계를 맺고 있다면, **Props를 우선적으로 사용 하고,** 여러 곳에서 공통으로 참조해야 하는 배경 데이터라면 **Provide/Inject를 사용** 하는 것이 건강한 설계이다.

## 2. 컴포저블(Composables): 로직 재사용

Composition API의 가장 강력한 장점이다.

특정 기능을 수행하는 로직을 별도의 함수로 분리하여 여러 컴포넌트에서 재사용할 수 있다.

- **규칙:** `use`로 시작하는 함수명을 사용한다 (예: useMouse, useFetch).

- **장점:** 컴포넌트 내부가 비즈니스 로직으로 복잡해지는 것을 막고, 로직을 독립적으로 테스트할 수 있다.

```javascript
// useMouse.js (재사용 가능한 로직)
import { ref, onMounted, onUnmounted } from 'vue';

export function useMouse() {
  const x = ref(0);
  const y = ref(0);

  const update = (e) => {
    x.value = e.pageX;
    y.value = e.pageY;
  };

  onMounted(() => window.addEventListener('mousemove', update));
  onUnmounted(() => window.removeEventListener('mousemove', update));

  return { x, y };
}
```

<br>

### 💡 TIP: Composable은 상태를 어떻게 공유하는가?

컴포저블을 사용할 때 **"상태를 어디에 선언하느냐"** 에 따라 데이터가 공유될지, 독립적일지 결정된다.

- **독립적 상태 (함수 내부 선언):** 함수를 부를 때마다 새로운 공간이 생긴다. 컴포넌트마다 각자의 카운터를 가진다.

- **공유 상태 (함수 외부 선언):** 함수 바깥에 변수를 두면, 앱 전체에서 하나의 변수를 같이 쓴다. 앱 어디서든 똑같은 값을 공유한다.

```javascript
// 독립적 상태 (함수 내부 선언 시)
export function useCounter() {
  const count = ref(0); // 호출될 때마다 새로운 count 생성
  return { count };
}

// 공유 상태 (함수 외부 선언 시)
const sharedCount = ref(0);
export function useSharedCounter() {
  return { count: sharedCount }; // 모든 호출자가 같은 값을 공유
}
```

<br>

### 💡 컴포저블(Composable), 무조건 분리하는 것이 좋은가?

컴포저블은 '로직의 모듈화' 를 돕지만, 모든 로직을 쪼개서 사용하는 것이 능사는 아니다.

다음 기준을 참고하여 분리 여부를 결정하는 것이 좋다.

- **로직의 재사용성:**
  - 동일한 로직이 두 개 이상의 컴포넌트에서 반복되는가? 그렇다면 컴포저블로 분리하는 것이 좋다.
  - 만약 특정 컴포넌트에서만 사용되는 로직이라면, 굳이 외부 파일로 분리하지 않고 컴포넌트 내부에 두는 것이 오히려 좋을 수 있다.

- **컴포넌트의 단순화:**
  - 스크립트 코드가 너무 길어져서 '무슨 일을 하는지' 한눈에 파악하기 어렵다면, 컴포저블로 분리하는 것이 좋다.
  - 로직이 비즈니스 로직(데이터 처리, API 호출)과 UI 로직(DOM 이벤트, 템플릿 제어)으로 섞여 있다면, 비즈니스 로직을 컴포저블로 분리하여 컴포넌트를 단순화 하는 것이 좋다.

- **테스트 가능성:**
  - 특정 로직을 컴포넌트와 분리해서 단독으로 테스트하고 싶은가?
  - 컴포저블로 만들면 순수 함수처럼 테스트하기 훨씬 수월해진다.

<br>

**❓ 언제 무엇을 선택할까?**

- **컴포넌트 내부에 유지:** 특정 컴포넌트에서만 쓰이는 간단한 로직
- **컴포저블로 분리:** 여러 컴포넌트에서 공통으로 호출하는 기능, 컴포넌트의 가독성을 해칠 정도로 방대한 로직

요약하면, 컴포저블은 **재사용성과 관심사 분리** 를 위한 도구이다.

단순히 코드를 쪼개는 것에 매몰되지 말고, **이 로직이 다른 곳에서도 쓰일 가능성이 있는가?** 를 먼저 고민하는 것이 좋다.

<br><br>

## 📚 정리

- **Provide/Inject** 는 복잡한 부모-자식 연결망을 건너뛰어 데이터를 직접 전달하는 '데이터 전용 통신망' 이다.

- **컴포저블** 은 반복되는 로직을 '레고 블록' 처럼 독립적인 함수로 만들어 코드 재사용성을 높이는 패턴이다.

- **핵심:** 애플리케이션의 규모가 커질수록 컴포저블을 활용해 로직을 잘게 쪼개고, 전역적인 데이터는 `Provide/Inject`나 `Pinia`를 통해 효율적으로 관리해야 한다.
