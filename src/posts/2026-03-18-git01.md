## 버전 관리란 무엇인가?

버전 관리는 파일의 변경 이력을 저장하고 관리하는 것이다.

쉽게 말해서,

> "언제든지 과거 상태로 돌아갈 수 있도록 기록을 남기는 시스템"

이라고 생각하면 된다.

예를 들어, 이런 상황을 생각해보면,

- 어제까지 잘 되던 코드가 오늘 갑자기 오류가 난다
- 여러 명이 동시에 파일을 수정해서 충돌이 난다

이럴 때 버전 관리가 없다면 직접 파일을 복사해두거나, 이름을 바꿔가며 관리해야 한다.
하지만 이는 매우 번거롭고, 실수하기 쉽다.

버전 관리 시스템을 사용하면 다음과 같은 기능을 쉽게 사용할 수 있다.

- 변경 이력 저장
- 이전 버전으로 복구
- 여러 사람이 동시에 작업 가능
- 누가 무엇을 수정했는지 추적

위의 작업을 수동으로 작업하는 것은 정말 쉽지 않을 것이다.

이러한 문제를 해결하기 위해 사용하는 도구가 바로 **Git**이다.

<br><br>

## Git을 사용하는 이유

Git을 사용하는 이유는 단순히 "편해서"가 아니다.

개발에서 필수적인 문제들을 해결해주기 때문이다.

<br>

1️⃣ **안전한 백업** <br>

Git은 모든 변경 이력을 저장한다.

따라서 문제가 생기더라도 언제든지 이전 상태로 되돌릴 수 있다.

<br>

2️⃣ **협업 가능** <br>

여러 개발자가 동시에 같은 프로젝트를 수정할 수 있다.

- 각자 작업 후 병합(merge)
- 충돌 발생 시 Git이 충돌 지점을 명확히 표시

즉, 협업에서 발생하는 혼란을 구조적으로 해결해준다.

<br>

3️⃣ **실험 가능** <br>

Git에서는 브랜치를 통해 독립적인 작업 공간을 만들 수 있다.

- 새로운 기능 개발
- 버그 수정
- 리팩토링

이 모든 작업을 기존 코드에 영향을 주지 않고 진행할 수 있다.

<br><br>

## Git은 어떻게 동작할까?

Git을 제대로 이해하려면 내부 동작 방식을 아는 것이 중요하다.

중요한 핵심들은 다음과 같다.

<br>

### 1. Git의 저장 방식: 스냅샷(Snapshot)

Git은 파일의 변경 내용(diff)이 아니라, 특정 시점의 전체 상태를 스냅샷으로 저장한다.

<img src="/images/git-snapshot.jpg" alt="스냅샷" width="600">

즉, 커밋은 단순한 변경 기록이 아니라 **그 시점의 프로젝트 전체 상태**를 의미한다.

<br>

**✅ 핵심 특징**

- **객체 기반 저장**
  - Git은 변경 사항들을 `commit`이라는 객체로 저장한다.
  - 각 커밋은 **SHA-1 해시값**을 고유 식별자로 가지며, 이를 통해 데이터의 무결성을 보장한다.

- **중복 데이터 최적화**
  - 파일의 내용이 이전 버전과 비교하여 수정되지 않았을 경우, Git은 새로운 데이터를 생성하지 않는다.
  - 그림 내의 점선 박스와 같이 이전 버전의 파일에 대한 **링크**만을 저장하여 저장 용량을 최적화한다.

- **독립적인 버전 관리 & 빠른 속도**
  - 각 버전은 이전 상태에 의존하여 계산할 필요가 없는 독립적인 스냅샷이다.
  - 따라서 특정 시점으로의 전환이나 브랜치 이동이 매우 신속하게 이루어진다.

<br>

### 2. Git 동작 구조

Git은 파일을 다음 4단계를 거쳐 관리한다.

<img src="/images/git-structure.jpg" alt="스냅샷" width="600">

1. **Working directory** : 로컬에서 실제로 작업을 하는 공간
2. **Staging area** : 커밋을 하기 전 임시 저장 공간
3. **Local repository** : `.git` 디렉토리에 스냅샷들이 물리적으로 저장되는 내 컴퓨터의 저장소
4. **Remote repository** : 원격 저장소. 웹 호스팅 서비스(ex. GitHub)에 생성한 프로젝트. 로컬 저장소와 동기화하여 사용

<br>

🔄 주요 데이터 흐름

1. 코드를 올리는 과정 (Local → Remote)
   - `add` : 워킹 디렉토리의 변경분을 스테이징 영역으로 이동 (커밋할 후보 지정)
   - `commit` : 스테이징 영역의 파일들을 하나의 스냅샷으로 묶어 로컬 저장소에 기록. 이 시점에 고유한 해시값이 생성된다.
   - `push` : 로컬 저장소의 커밋 객체들을 원격 저장소로 전송하여 공유한다.

2. 코드를 가져오는 과정 (Remote → Local)
   - `clone` : 원격 저장소의 모든 내용을 처음으로 로컬에 복제하여 로컬 저장소를 생성한다.
   - `fetch` : 원격 저장소의 최신 변경 사항들을 확인만 하고 로컬 저장소로 가져온다. (내 작업 코드와 합치진 않음)
   - `pull` : 원격 저장소의 코드를 가져와서(`fetch`) 현재 내 **Working directory**에 즉시 합친다.(`merge`)
   - `merge` : 로컬 내에서 서로 다른 브랜치나 가져온 변경 사항을 **Working directory**의 코드와 병합한다.

<br>

### 3. 분산 버전 관리 시스템

Git의 또 다른 특징은 중앙 서버에 의존하지 않는다는 것이다.

즉, 모든 개발자가 저장소 전체를 복사해서 가지고 있다.

<br>

**✅ 이 구조의 장점**

- 서버가 다운되어도 복구가 가능
- 오프라인에서도 작업 가능
- 모든 로컬 저장소가 백업 역할 수행

<br><br>

## 정리

- 버전 관리는 파일의 변경 이력을 기록하고 언제든지 과거로 돌아갈 수 있게 해준다.
- Git은 이러한 버전 관리를 효율적으로 수행하기 위한 분산 버전 관리 시스템이다.
- Git은 변경 내용이 아닌 프로젝트 전체 상태를 스냅샷으로 저장한다.
- 각 커밋은 고유한 해시값을 가지며 데이터의 무결성을 보장한다.
- Git은 `Working Directory → Staging Area → Local Repository → Remote Repository`의 흐름으로 동작한다.
- Git은 모든 개발자가 전체 저장소를 가지는 분산 구조를 가진다.
