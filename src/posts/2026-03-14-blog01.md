## Vue + Vite로 개발 블로그 만들기 (1)

원래는 **Jekyll + GitHub Pages** 기반으로 블로그를 운영하고 있었다.
GitHub Pages와 잘 맞고 정적 블로그를 빠르게 만들 수 있다는 장점이 있어서 사용하게 되었다.

하지만 Jekyll은 Ruby 기반이다 보니 종종 불편한 점이 있었다.
Ruby와 Jekyll의 버전을 맞춰야 하거나, gem 충돌이 발생하는 경우가 있었고
환경 설정을 다시 맞추는 데 시간을 쓰는 일이 생기곤 했다.

최근에 Vue를 공부하게 되면서,
이왕이면 블로그도 직접 만들어보면 좋겠다는 생각이 들었다.

그래서 이번 기회에 **Vue + Vite 기반으로 직접 정적 블로그를 만들어보기로 했다.**

<br>

이번 글에서는 아래 과정까지 진행했다.

- Vue + Vite 프로젝트 생성
- Vue Router 설정
- Markdown 기반 블로그 구조 설계
- Github Pages 배포

> 모든 과정은 **Mac 환경**에서 진행했습니다.

<br><br>

## 1. 프로젝트 생성

먼저 **Vite**를 이용해 Vue 프로젝트를 생성했다.

Vite는 빠른 개발 서버와 간단한 설정을 제공하는 프론트엔드 빌드 도구로 최근 Vue 프로젝트에서 많이 사용되는 방식이다.

다음 명령어를 실행하면 프로젝트를 생성할 수 있다.

```bash
npm create vite@latest
```

명령어를 실행하면 몇 가지 선택 옵션이 나오는데

- 프로젝트 이름 입력
- Framework 선택 → **Vue**
- Variant 선택 → **JavaScript**

위 순서로 진행했다.

<br>

프로젝트 생성 후 의존성을 설치한다.

```bash
npm install
```

설치가 완료되면 다음 명령어를 통해 로컬 서버를 실행할 수 있다.

```bash
npm run dev
```

기본적으로 아래 주소를 통해 기본 페이지를 확인할 수 있다.

```
http://localhost:5173
```

Vite의 장점 중 하나는 개발 서버가 매우 빠르게 실행된다는 점이다.

또한 파일을 수정하면 즉시 화면에 반영되는 HMR(Hot Module Replacement) 기능도 제공한다.

<br><br>

## 2. Vue Router 설치

> Vue는 기본적으로 **SPA(Single Page Application)** 구조로 동작한다.

SPA에서는 페이지를 이동할 때 서버에서 새로운 HTML을 받아오는 것이 아니라
하나의 페이지 안에서 화면만 변경하는 방식으로 동작한다.

이때 URL에 따라 어떤 컴포넌트를 보여줄지 관리하는 라이브러리가 바로 **Vue Router** 이다.

예를 들어

- `/` → 홈 화면

- `/post/slug` → 특정 블로그 글

과 같이 URL에 따라 다른 화면을 보여줄 수 있다.

<br>

먼저 Vue Router를 설치한다.

```bash
npm install vue-router
```

이후 프로젝트에 `src/router/index.js` 파일을 생성하고 다음과 같이 라우터를 설정했다.

현재는

- 홈 페이지
- 블로그 글 상세 페이지

두 개의 라우트만 간단하게 구성했다.

블로그 기능이 늘어나면 라우팅 구조도 조금 더 확장할 예정이다.

```js
import { createRouter, createWebHashHistory } from 'vue-router';
import Home from '../pages/Home.vue';
import PostDetail from '../pages/PostDetail.vue';

const router = createRouter({
  history: createWebHashHistory(),
  routes: [
    {
      path: '/',
      component: Home,
    },
    {
      path: '/post/:slug',
      component: PostDetail,
    },
  ],
});

export default router;
```

<br>

Vue Router는 브라우저의 **History API**를 이용해 URL 변경을 관리한다.

이때 어떤 방식으로 URL 히스토리를 관리할지에 따라 `createWebHistory`와 `createWebHashHistory` 두 가지 방식을 사용할 수 있다.

- `createWebHistory` <br>
  : 일반적인 URL 방식 (`/post/slug`)

- `createWebHashHistory` <br>
  : URL에 `#`이 포함되는 방식 (`/#/post/slug`)

<br>

여기서는 `createWebHashHistory` 방식을 사용했다.

그 이유는 GitHub Pages 환경 때문이다.

GitHub Pages는 기본적으로 정적 파일 서버이기 때문에 SPA 라우팅을 직접 처리하지 못한다.

예를 들어 `createWebHistory`를 사용하면

```bash
example.com/post/vue
```

와 같은 URL에 직접 접근했을 때

서버는 `/post/vue` 경로에 해당하는 파일을 찾게 된다.

하지만 GitHub Pages와 같은 정적 호스팅 환경에서는

`index.html` 외에 `/post/vue`에 해당하는 실제 파일이 존재하지 않는다.

그래서 해당 경로를 처리하지 못하고 **404 Not Found 오류가 발생할 수 있다.**

반면, `createWebHashHistory` 방식은 URL의 `#` 뒤에 있는 경로를 브라우저에서만 처리한다.

예를 들어

```
example.com/#/post/vue
```

위의 경우 서버에는 항상 `/` 요청만 전달되고, `#` 뒤의 경로는 Vue Router가 브라우저에서 처리한다.

그래서 GitHub Pages와 같은 정적 호스팅 환경에서도 라우팅 문제 없이 안정적으로 동작한다.

## 3. 블로그 구조 설계

먼저 프로젝트 구조를 정리해 보았다.

정적 블로그이기 때문에 크게 보면 다음 세 가지 요소가 필요하다.

- 페이지 컴포넌트 (홈, 글 상세 등)
- 마크다운 글 저장소
- 라우팅 및 데이터 처리 로직

이를 기준으로 프로젝트 구조를 다음과 같이 구성했다.

```
.github
 └ workflows
    └ deploy.yml

src
 ├ assets
 │   └ css
 ├ components
 ├ pages
 │   ├ Home.vue
 │   └ PostDetail.vue
 ├ posts
 │   └ example.md
 ├ router
 │   └ index.js
 ├ utils
 │   └ posts.js

App.vue
main.js
package.json
```

각 중요 폴더의 역할을 간단히 정리하면 다음과 같다.

<br>

1️⃣ `.github`

```
.github/workflows/deploy.yml
```

이 폴더는 GitHub Actions 설정 파일을 저장하는 곳이다.

`deploy.yml` 파일을 통해

- main 브랜치에 코드가 push되면
- 프로젝트를 자동으로 build하고
- GitHub Pages에 배포

되도록 설정했다.

즉, 코드를 push하기만 하면 블로그가 자동으로 배포되는 구조다.

<br>

2️⃣ `components`

```
src/components
```

여기에는 재사용 가능한 Vue 컴포넌트를 넣는다.

예를 들어

- Header
- Footer
- Navigation

같은 컴포넌트를 만들 수 있다.

페이지에서 반복적으로 사용하는 UI를 이곳에 모아서 사용한다.

<br>

3️⃣ `pages`

```
src/pages
```

이 폴더에는 라우터와 연결되는 페이지 컴포넌트가 들어간다.

현재는 두 개의 페이지가 존재한다.

- `Home.vue` → 블로그 글 목록을 보여주는 페이지
- `PostDetail.vue` → 특정 블로그 글을 보여주는 페이지

라우터에서 URL에 따라 이 페이지들이 렌더링된다.

<br>

4️⃣ `posts`

```
src/posts
```

이 폴더에는 블로그 글을 마크다운 파일로 저장한다.

파일 내부 구조는 다음과 같이 이루어져 있다.

```
---
title: 첫 번째 글
summary: 블로그 테스트 글
date: 2026-03-14
slug: first-post
---

# Hello Blog

첫 번째 테스트 글입니다.

```

위의 `---`로 감싸진 영역은 **frontmatter**라고 한다.

여기에는 글에 대한 메타 데이터를 작성한다.

예를 들어

- `title` → 글 제목
- `summary` → 글 요약
- `date` → 작성 날짜
- `slug` → URL에 사용될 값

이 정보들은 글 목록을 만들거나 라우팅을 처리할 때 사용된다.

<br><br>

## 4. 마크다운 글 불러오기

앞에서 블로그 글을 마크다운 파일로 관리하는 구조를 만들었다.

```
src/posts
 └ example.md
```

이제 Vue 애플리케이션에서 이 마크다운 파일들을 읽어서 블로그 글 목록과 글 상세 페이지에 렌더링해야 한다.

이를 위해 다음 작업이 필요하다.

- `posts` 폴더에 있는 마크다운 파일을 가져오기
- 파일 안의 `frontmatter` 메타 데이터 추출
- 마크다운을 HTML로 변환

이 프로젝트에서는 Vite의 `import.meta.glob` 기능을 이용해

`posts` 폴더의 마크다운 파일을 한 번에 불러오도록 구현했다.

또한 `marked` 라이브러리를 이용해 마크다운을 HTML로 변환하도록 구성했다.

<br><br>

## 5. GitHub Pages 배포

배포는 **GitHub Pages**를 사용했다.

GitHub Pages를 선택한 이유는 다음과 같다.

- 무료로 정적 사이트 호스팅 가능
- GitHub 레포지토리와 연동이 쉬움
- GitHub Actions를 이용해 자동 배포 가능

프로젝트를 깃허브 레포에 올린 뒤 GitHub Actions를 이용해 main 브랜치에 push 되면 자동으로 배포되도록 설정했다.

```
.github/workflows/deploy.yml
```

```yaml
name: Deploy Vue to GitHub Pages

on:
  push:
    branches:
      - main

permissions:
  contents: write

jobs:
  deploy:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout repository
        uses: actions/checkout@v4

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install dependencies
        run: npm install

      - name: Build project
        run: npm run build

      - name: Deploy to GitHub Pages
        uses: peaceiris/actions-gh-pages@v4
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
```

간단히 흐름을 정리하면 다음과 같다.

1. main 브랜치에 코드 push
2. GitHub Actions 실행
3. 프로젝트 build (`npm run build`)
4. `dist` 폴더를 GitHub Pages에 배포

이렇게 설정해 두면 코드를 push 하는 것만으로 자동 배포가 이루어진다.
